# 0010: i18n — `lang` attribute and overridable UI strings

Date: 2026-07-18
Status: accepted

## Context

Issue #7 (WCAG 3.1.1): the generated HTML has no `lang` attribute, and every
fixed UI string is hard-coded English (required errors, submit
failed/succeeded, the Submit button, `Comment — {row}`, the required legend,
the submitting indicator). A Japanese form currently shows English text in
all of those places.

Two schema surfaces were considered:

1. A `lang` field plus per-message overrides (`messages.required`, …).
2. Built-in translation bundles selected by `lang` alone.

Bundles alone can never cover every language; overrides alone force authors
to re-type every string even for languages we could ship.

## Decision

- **Both, layered.** `lang` (top-level, BCP 47-ish tag, default `"en"`)
  selects a built-in bundle; an optional top-level `messages` map overrides
  individual strings on top of it. Built-in bundles: `en` and `ja`. An
  unknown `lang` falls back to the `en` strings but is still emitted as
  `<html lang="…">` verbatim (authors then override via `messages`).
- **One flat `messages` map** (no `labels`/`messages` split), strict keys —
  a typo is a generation error:

  | Key               | Placeholders       | `en` default                            |
  | ----------------- | ------------------ | --------------------------------------- |
  | `required`        | `{title}`          | `"{title}" is required.`                |
  | `required_row`    | `{row}`, `{title}` | `"{row}" in "{title}" is required.`     |
  | `required_legend` | `{mark}`           | `{mark} indicates required`             |
  | `submit`          | —                  | `Submit`                                |
  | `submitting`      | —                  | `Submitting…`                           |
  | `submit_failed`   | —                  | `Submission failed. Please try again.`  |
  | `submit_success`  | —                  | `Your response has been submitted.`     |
  | `comment`         | `{row}`            | `Comment — {row}`                       |

- **Placeholder syntax**: `{name}` interpolation (`{title}` = item title,
  `{row}` = table/rubric row title, `{mark}` = the rendered `*` required
  mark). Unknown placeholders are left literal; there is no escape syntax.
- **Precedence for the success message**: `post_submit.message` >
  `messages.submit_success` > bundle. `post_submit.message` predates this
  decision and keeps working unchanged.
- **One shared module** (`src/messages.ts`) holds the key list, bundles,
  resolution (`lang` → bundle → overrides), and interpolation. The generator
  uses it at generation time (button label, legend, comment labels); the
  runtime imports the same module and resolves from the embedded form JSON,
  which already carries `lang` and `messages`.

## Consequences

- Existing English forms render byte-identical strings; the only output
  change is the (previously missing) `lang="en"` attribute.
- Adding a language means adding one bundle to `src/messages.ts`; authors of
  unshipped languages set `lang` plus a full `messages` map.
- The `{row}: {choice}` cell accessible-name separator stays hard-coded; it
  is punctuation, not prose, and can join the message table later if needed.
- JSON Schema gains `lang` + `messages`, so editors autocomplete the keys.
